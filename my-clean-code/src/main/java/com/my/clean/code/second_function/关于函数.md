

## 最重要的DRY原则：不要重复

### 1、一个方法只干一件事！

### 2、一个方法不要太长

### 3、一个函数只处理当前这个分层的逻辑

具体意思就是：
- mapper 层面的函数，专注于数据库数据组装
- service 层面的函数，专注于业务处理
- controller 层面的函数，专注于返回值的构建

不要在service层面做不属于这个层面的逻辑，比如构建返回值等！
如此才能保证代码的清晰。


### 4、一般来说，函数的参数越少表示函数的复杂度越低

- 前提是不处理全局变量这种东西。

参数越多，表示当前函数需要依赖的条件就越多，逻辑就越复杂，这是显而易见的
这也是函数式编程提倡的

当参数超过3个，就要把参数封根据情况装成类，吞掉复杂的依赖关系，是代码变得清晰易读



### 5、函数是无副作用的，幂等的

也就是说，函数无论调用多少次，其结果都是可预期的，而不是无法预测的

另外：

- dealUserState(user) 这种方法就是典型的改变了入参的某些字段，使得入参变了

如何解决？

- user.dealUserState() 这种方式，将user内部属性处理逻辑封装到user内部，就非常直观知道这个方法要干什么


### 6、try/catch代码抽出单独的方法

能够使得代码的可读性变好
还能分别处理异常情况


## 实现方式：先思考设计，再写，最后不断重构到满意为止
